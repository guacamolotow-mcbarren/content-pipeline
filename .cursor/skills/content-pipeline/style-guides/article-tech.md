# Style Guide: Технические статьи (IT-площадки, корпблоги)

## Формат
- Техническая / экспертная статья для IT-площадки или корпоративного блога
- Типы: «Обзор», «Кейс», «Мнение» — указывать в метаданных
- Длина: 8000–20000 знаков (8–15 мин чтения)
- Требования площадки (теги, хабы, формат) — уточнять в брифе

## Тон
- «Всем привет!» — стандартное открытие для корпблога
- Экспертный, но живой; допустимы спойлеры, самоирония, отсылки к мемам IT-сообщества
- Баланс: 60% польза / 40% storytelling

## Структура

```markdown
# [Заголовок: конкретная проблема или результат]

[Лид: приветствие, кто автор, зачем читать, спойлер если уместен]

## [H3 — контекст / проблема]
[Постановка, почему это важно]

## [H3 — подход / решение]
[Основной контент: схемы, код, таблицы, списки]

## [H3 — результаты / выводы]
[Что получилось, уроки, рекомендации]

### Кому полезно
[Целевая аудитория: роли, кому стоит прочитать]
```

## Обязательные элементы
- Представление: «Меня зовут [Имя], я [роль] в компании «Инфосистемы Джет»»
- Подзаголовки уровня `###` (H3), не H2
- Минимум 1 схема, таблица, блок кода или нумерованный список
- Ссылки на связанные материалы (другие статьи Jet, внешние источники)
- Блок «Кому полезно» перед финалом (для длинных статей)

## Технические детали
- Код в блоках с указанием языка
- Длинные листинги — с комментариями к ключевым строкам
- Диаграммы: ASCII, PlantUML, или описание для вставки
- Таблицы Markdown для сравнений

## Лексика
- IT-термины без упрощения для технической аудитории
- Допустимы английские термины (SDLC, MVP, threat model) без перевода
- Списки с `-` для перечислений, нумерация для шагов

## Запреты
- Рекламные формулировки («лучшее решение на рынке»)
- Статья без технической глубины (это не пресс-релиз)
- Копипаст из документации без адаптации
